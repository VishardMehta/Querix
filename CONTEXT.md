# CONTEXT.md — Full Backend Specification for "Talk to Data"

## What This Is

You are building the complete backend for a hackathon project called "Talk to Data" — a multi-agent AI system that lets non-technical users ask natural language questions about data and get accurate answers. The system handles structured queries (SQL), unstructured queries (RAG over documents), and hybrid queries (both combined).

**Tech Stack (all free-tier, no paid services):**
- Python 3.10+
- Groq API (free tier) — Llama 3.1 8B Instant + Llama 3.3 70B Versatile
- SQLite (database)
- FAISS (vector search, in-memory)
- sentence-transformers (embeddings — `all-MiniLM-L6-v2`)
- Streamlit (frontend)
- Plotly (charts)
- PyYAML (semantic layer config)

**Important constraints:**
- No LangChain, no CrewAI, no frameworks. Use raw Groq SDK calls.
- No paid APIs. Everything must work with Groq free tier.
- No external databases. SQLite file + FAISS in-memory only.
- Keep it simple. Each agent is a Python function, not a class hierarchy.

### 1. `app/core/groq_client.py`

Shared Groq client. All agents import from here.

```python
import os
from groq import Groq

client = Groq(api_key=os.getenv("GROQ_API_KEY"))

MODELS = {
    "fast": "llama-3.1-8b-instant",      # Router, fast tasks
    "smart": "llama-3.3-70b-versatile",   # SQL gen, answer gen
}

def call_llm(model_key: str, system_prompt: str, user_message: str, temperature: float = 0.0, json_mode: bool = False):
    """Single LLM call. Returns string (or parsed dict if json_mode)."""
    kwargs = {
        "model": MODELS[model_key],
        "messages": [
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_message}
        ],
        "temperature": temperature,
    }
    if json_mode:
        kwargs["response_format"] = {"type": "json_object"}
    
    response = client.chat.completions.create(**kwargs)
    text = response.choices[0].message.content
    
    if json_mode:
        import json
        return json.loads(text)
    return text
```

### 3. `config/verified_queries.yaml`

Golden question→SQL pairs for few-shot retrieval. Write at least 3 per pattern.

```yaml
verified_queries:
  # ===== CHANGE ANALYSIS =====
  - question: "Why did revenue drop last month?"
    pattern: CHANGE_ANALYSIS
    sql: |
      SELECT 
        strftime('%Y-%m', order_date) as month,
        SUM(amount) as revenue
      FROM orders
      WHERE order_date >= date('now', '-2 months')
        AND status = 'completed'
      GROUP BY month
      ORDER BY month

  - question: "What caused the increase in complaints?"
    pattern: CHANGE_ANALYSIS
    sql: |
      SELECT 
        strftime('%Y-%m', created_date) as month,
        category,
        COUNT(*) as complaint_count
      FROM complaints
      WHERE created_date >= date('now', '-3 months')
      GROUP BY month, category
      ORDER BY month, complaint_count DESC

  - question: "Why did orders decrease in the South region?"
    pattern: CHANGE_ANALYSIS
    sql: |
      SELECT 
        strftime('%Y-%m', order_date) as month,
        region,
        COUNT(*) as order_count,
        SUM(amount) as revenue
      FROM orders
      WHERE region = 'South'
        AND order_date >= date('now', '-3 months')
      GROUP BY month
      ORDER BY month

  # ===== COMPARISON =====
  - question: "Compare North vs South region revenue"
    pattern: COMPARISON
    sql: |
      SELECT 
        region,
        SUM(amount) as total_revenue,
        COUNT(*) as total_orders,
        AVG(amount) as avg_order_value
      FROM orders
      WHERE region IN ('North', 'South')
        AND status = 'completed'
      GROUP BY region

  - question: "Compare this month vs last month sales"
    pattern: COMPARISON
    sql: |
      SELECT 
        CASE 
          WHEN order_date >= date('now', 'start of month') THEN 'This Month'
          ELSE 'Last Month'
        END as period,
        SUM(amount) as revenue,
        COUNT(*) as orders
      FROM orders
      WHERE order_date >= date('now', 'start of month', '-1 month')
        AND status = 'completed'
      GROUP BY period

  - question: "Compare online vs retail channel performance"
    pattern: COMPARISON
    sql: |
      SELECT 
        channel,
        SUM(amount) as revenue,
        COUNT(*) as orders,
        AVG(amount) as avg_order
      FROM orders
      WHERE status = 'completed'
      GROUP BY channel

  # ===== BREAKDOWN =====
  - question: "What makes up total revenue by category?"
    pattern: BREAKDOWN
    sql: |
      SELECT 
        p.category,
        SUM(o.amount) as revenue,
        ROUND(SUM(o.amount) * 100.0 / (SELECT SUM(amount) FROM orders WHERE status='completed'), 1) as pct
      FROM orders o
      JOIN products p ON o.product_id = p.product_id
      WHERE o.status = 'completed'
      GROUP BY p.category
      ORDER BY revenue DESC

  - question: "Show the breakdown of complaints by category"
    pattern: BREAKDOWN
    sql: |
      SELECT 
        category,
        COUNT(*) as count,
        ROUND(COUNT(*) * 100.0 / (SELECT COUNT(*) FROM complaints), 1) as pct
      FROM complaints
      GROUP BY category
      ORDER BY count DESC

  - question: "Breakdown of orders by region"
    pattern: BREAKDOWN
    sql: |
      SELECT 
        region,
        COUNT(*) as orders,
        SUM(amount) as revenue,
        ROUND(SUM(amount) * 100.0 / (SELECT SUM(amount) FROM orders), 1) as pct
      FROM orders
      GROUP BY region
      ORDER BY revenue DESC

  # ===== SUMMARY =====
  - question: "Give me a summary of this month's metrics"
    pattern: SUMMARY
    sql: |
      SELECT 
        SUM(amount) as total_revenue,
        COUNT(*) as total_orders,
        AVG(amount) as avg_order_value,
        COUNT(DISTINCT customer_id) as unique_customers,
        SUM(CASE WHEN status='cancelled' THEN 1 ELSE 0 END) as cancelled_orders,
        SUM(CASE WHEN status='returned' THEN 1 ELSE 0 END) as returned_orders
      FROM orders
      WHERE order_date >= date('now', 'start of month')

  - question: "Weekly summary of customer metrics"
    pattern: SUMMARY
    sql: |
      SELECT
        strftime('%W', order_date) as week,
        COUNT(DISTINCT customer_id) as active_customers,
        COUNT(*) as orders,
        SUM(amount) as revenue,
        AVG(amount) as avg_order
      FROM orders
      WHERE order_date >= date('now', '-28 days')
      GROUP BY week
      ORDER BY week

  # ===== GENERAL =====
  - question: "Top 5 customers by spending"
    pattern: GENERAL
    sql: |
      SELECT 
        c.name,
        c.segment,
        SUM(o.amount) as total_spent,
        COUNT(o.order_id) as order_count
      FROM orders o
      JOIN customers c ON o.customer_id = c.customer_id
      WHERE o.status = 'completed'
      GROUP BY c.customer_id, c.name, c.segment
      ORDER BY total_spent DESC
      LIMIT 5

  - question: "Which products have the highest return rate?"
    pattern: GENERAL
    sql: |
      SELECT 
        p.name,
        p.category,
        COUNT(CASE WHEN o.status='returned' THEN 1 END) as returns,
        COUNT(*) as total_orders,
        ROUND(COUNT(CASE WHEN o.status='returned' THEN 1 END) * 100.0 / COUNT(*), 1) as return_rate
      FROM orders o
      JOIN products p ON o.product_id = p.product_id
      GROUP BY p.product_id, p.name, p.category
      HAVING total_orders > 5
      ORDER BY return_rate DESC
      LIMIT 10
```

### 6. `app/core/cache.py`

Semantic cache using embeddings.

```python
"""
Semantic cache implementation.

Methods:
- find_similar(question: str, threshold: float = 0.90) → cached result or None
- store(question: str, result: dict) → None
- clear() → None

Each cache entry stores:
{
    "question": str,
    "embedding": numpy array,
    "sql": str (or None for RAG),
    "results": any,
    "answer": str,
    "tables_used": list,
    "timestamp": datetime,
    "response_time_ms": int
}

Use sentence_transformers SentenceTransformer('all-MiniLM-L6-v2') for embeddings.
Use cosine similarity (numpy dot product on normalized vectors).
TTL: 1 hour (configurable but not important for demo).
```

## Main Pipeline Flow (in main.py)

```python
def process_query(question: str) -> dict:
    start_time = time.time()
    
    # 1. Cache check
    cached = cache.find_similar(question)
    if cached:
        elapsed = (time.time() - start_time) * 1000
        return {**cached, "cached": True, "time_ms": elapsed}
    
    # 2. Route
    schema_summary = semantic_layer.get_schema_summary()
    state_context = conversation_state.get_context_for_followup()
    route = router.route(question, schema_summary, state_context)
    
    if route["intent"] == "OUT_OF_SCOPE":
        return {
            "answer": "I can answer questions about orders, customers, products, and complaints. This question is outside my data scope.",
            "cached": False, "time_ms": (time.time()-start_time)*1000
        }
    
    # 3. Process based on intent
    sql_results = None
    rag_docs = None
    sql_used = None
    
    if route["intent"] in ["STRUCTURED", "HYBRID"]:
        # Get relevant tables
        relevant_tables = vector_store.search_tables(question, top_k=5)
        semantic_context = semantic_layer.generate_ddl_with_context(relevant_tables)
        
        # Find best verified query
        verified = vector_store.find_similar_query(question, route["pattern"])
        
        # Generate SQL
        gen_result = sql_generator.generate_sql(
            question, route["pattern"], semantic_context, verified, state_context
        )
        sql_used = gen_result["sql"]
        
        # Validate and execute (with retry)
        exec_result = validator.validate_and_execute(sql_used, semantic_layer, "data/demo.db")
        
        if not exec_result["success"] and exec_result["error_type"] in ["SCHEMA", "SYNTAX"]:
            # Retry once with error feedback
            gen_result = sql_generator.generate_sql(
                question, route["pattern"], semantic_context, verified, state_context,
                error_feedback=exec_result["error"]
            )
            sql_used = gen_result["sql"]
            exec_result = validator.validate_and_execute(sql_used, semantic_layer, "data/demo.db")
        
        if not exec_result["success"]:
            return {
                "answer": f"I couldn't generate a valid query for this question. {exec_result['error']}",
                "error": True, "time_ms": (time.time()-start_time)*1000
            }
        
        sql_results = exec_result
    
    if route["intent"] in ["UNSTRUCTURED", "HYBRID"]:
        rag_docs = vector_store.search_documents(question, top_k=5)
    
    # 4. Generate answer
    answer_result = answer_generator.generate_answer(
        question, route["pattern"], sql_results, rag_docs, sql_used, state_context
    )
    
    # 5. Update state
    tables_used = gen_result.get("tables_used", []) if sql_results else []
    conversation_state.update(question, sql_used, sql_results, answer_result["answer"], tables_used)
    
    # 6. Build response
    elapsed = (time.time() - start_time) * 1000
    response = {
        "answer": answer_result["answer"],
        "sql": sql_used,
        "results": sql_results,
        "rag_docs": rag_docs,
        "tables_used": tables_used,
        "confidence": gen_result.get("confidence", None) if sql_results else None,
        "follow_ups": answer_result.get("follow_up_questions", []),
        "chart": answer_result.get("chart_suggestion", None),
        "route": route,
        "cached": False,
        "time_ms": elapsed
    }
    
    # 7. Cache the response
    cache.store(question, response)
    
    return response
```

## Critical Implementation Notes

1. **Groq rate limits on free tier:** Llama 3.3 70B has 30 RPM and ~1000 RPD. Each full query uses 2 calls (SQL gen + answer gen). Plan for ~500 queries per day max. The router uses 8B which has separate, higher limits.

2. **SQLite date functions:** Use `date()`, `strftime()`, `date('now')`, `date('now', '-1 month')`, `date('now', 'start of month')`. Do NOT use `DATEADD`, `DATEDIFF`, `NOW()` — those are not SQLite.

3. **Embeddings are loaded once:** Use `@st.cache_resource` in Streamlit to load the SentenceTransformer model and FAISS indexes once, not on every request.

4. **SQL cleaning is essential:** LLMs often wrap SQL in ```sql``` blocks or add explanatory text. The `clean_sql()` function must strip ALL of this before execution.

5. **Error messages to LLM must be specific:** Don't just say "error." Say "Column 'revenue_total' does not exist in table 'orders'. Available columns: order_id, customer_id, amount, status, region, channel, order_date". The more specific, the better the retry.

6. **JSON mode in Groq:** Use `response_format={"type": "json_object"}` for router, SQL generator, and answer generator. Always include "Return JSON" in the system prompt when using JSON mode.

7. **Chart generation can fail gracefully:** If chart_suggestion is None or results can't be charted, just skip the chart. Never crash because of chart rendering.

8. **Conversation state is per-session:** Reset when user clicks "Clear conversation" or refreshes the page. Stored in `st.session_state`.
